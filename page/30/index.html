GIT

发表于 2017-06-21 | 分类于 运维笔记

基本概念

文件状态

文件在目录中的状态有两种：

版本状态

文件已经加入GIT版本控制时，一定有三种状态，文件一定处于其中一个：

  • 已提交
  • 已修改
  • 已暂存

三个区域

  • Repository：仓库区（或本地仓库）

Git 仓库目录是 Git 用来保存项目的元数据和对象数据库的地方。 这是 Git 中最重要的部分，从其它计算机克隆仓库时，拷贝的就是这里的数据。在工作区有个隐藏目录.git，这个不算工作区，而是 Git 的仓库。

  • Workspace：工作区

工作区域就是目前看到能操作的目录和文件，它们可以是从项目的某个版本独立提取出来的内容，这些从 Git 仓库的压缩数据库中提取出来的文件，放在磁盘上供你使用或修改的文件。

  • Index / Stage：暂存区

暂存区域是一个文件，保存了下次将提交的文件列表信息，一般在 Git 仓库目录中。 有时候也被称作 ’索引‘ ，不过一般说法还是叫暂存区域。

工作流程

  1. 在工作目录中修改文件。
  2. 暂存文件，将文件的快照放入暂存区域。
  3. 提交更新，找到暂存区域的文件，将快照永久性存储到 Git 仓库目录。

基础配置

配置文件

Git 自带一个 git config 的工具来帮助设置控制 Git 外观和行为的配置变量，这些变量存储在三个不同的位置：

  1. /etc/gitconfig
  1. ~/.gitconfig or ~/.config/git/config
  1. .git/config

每一个级别覆盖上一级别的配置，所以 .git/config 的配置变量会覆盖 /etc/gitconfig 中的配置变量，在 Windows 系统中，Git 会查找 $HOME 目录下（一般情况下是 C:\Users\$USER）的 .gitconfig 文件。 Git 同样也会寻找 /etc/gitconfig 文件，但只限于 MSys 的根目录下，即安装 Git 时所选的目标位置。

设置用户

当安装完 Git 应该做的第一件事就是设置你的用户名称与邮件地址。 这样做很重要，因为每一个 Git 的提交都会使用这些信息，并且它会写入到你的每一次提交中，不可更改：

再次强调，如果使用了 --global 选项，那么该命令只需要运行一次，因为之后无论你在该系统上做任何事情， Git 都会使用那些信息。 当你想针对特定项目使用不同的用户名称与邮件地址时，可以在那个项目目录下运行没有 --global 选项的命令来配置。

检查配置

如果想要检查你的配置，可以使用 git config --list 命令来列出所有 Git 当时能找到的配置，重复出现的配置则以最后一次的值为准。

git init 创建一个空的 Git 版本库或重新初始化一个已存在的版本库

提交变更

指令 作用
git commit -m 'msg' 仅提交暂存区的文件到版本库，-m 为提交添加说明信息

git commit --amend -m 'msg' 修订上一次的提交

撤销操作

命令 作用
git reset [file] 重置暂存区的指定文件，与上一次commit保持一致，但工作区文件不变
git reset --hard 重置暂存区与工作区，与上一次commit保持一致
git reset [commit] 重置当前分支的指针为指定commit，同时重置暂存区，但工作区不变
git reset --hard [commit] 重置当前分支的HEAD为指定commit，同时重置暂存区和工作区，与指定commit一致
git reset --keep [commit] 重置当前HEAD为指定commit，但保持暂存区和工作区不变

丢弃工作区

命令 作用

git checkout . 恢复暂存区的所有文件到工作区
git checkout [commit] [file] 恢复某个commit的指定文件到暂存区和工作区

HEAD

HEAD 是当前分支引用的指针，它总是指向该分支上的最后一次提交。 这表示 HEAD 将是下一次提交的父结点。 通常，理解 HEAD 的最简方式，就是将它看做 你的上一次提交 的快照。

当存在多个分支时，^与~之间的区别，例如：下面的图来具体感受一下：^与~的区别。

~只选择主分支，如：HEAD~2表示第 2 级父节点的第1个分支。

^可以选择分支，如：HEAD~2^3表示第 2 级父节点的第3个分支。

tornado.options

tornado 的命令行参数解析器，它的 define 方法负责定义接收的参数，options 则是接收并且解析完成后的参数，可以使用 options.parse_command_line 解析命令行参数，或者使用 options.parse_config_file 解析配置文件中的参数

  • 单应用项目文件组织
  • index.py 视图文件
  • config.py 配置文件
  • app.py 应用对象，视图和路由在这里配置
  • server.py 服务器程序文件

tornado 路由

路由 模式
(r”/“, index.MainHandler) 绝对路径匹配
(r”/“, index.MainHandler,{“key”: “value”}) 向视图传递参数，在视图函数的 initialize 方法中可以接收 key 这个参数，然后赋值 self.key = key
(r”/\d+”, index.MainHandler) 正则表达式匹配
tornado.web.url(r”/url”, index.Url, {“key1”: “value1”, “key2”: “value2”}, name=”urlfor”) 定义命名的路由条目，可以在视图中使用 self.reverse_url(name=urlfor)，获得匹配正则为 urlfor 的 url 地址
(r”/index/(?P\<page>\d*)”, index.MainHandler) 正则表达式分组匹配
(r”/request/(\w+)/(\d+)”, request.MainHandler),
